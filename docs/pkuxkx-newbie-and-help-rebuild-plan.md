# 1. 方案摘要

本方案以当前 `xkx2001-utf8` 的 FluffOS 服务端、Node Gateway、React Web 客户端和部署/测试体系为底座，引入 PKUXKX 的柳秀山庄新手村体验，并重塑为 Web 优先的新手成长与帮助体系。

核心决策：

- **新角色不再进入侠客岛**，统一进入 PKUXKX 的柳秀山庄新手村；
- 新手村教学期间四维固定为 `20 / 20 / 20 / 20`；
- 完成新手村、离开前，玩家可在 Web 中进行一次不可逆的四维重分配；
- 四维总点数固定为 80，每项范围为 10～30；
- 新手村教学武功、教学经验和教学物品在毕业时清除，保留毕业奖励和最终属性；
- 帮助内容以 PKUXKX 帮助资料为内部事实来源，但逐篇重写、核对当前实现后发布；
- 帮助系统仅提供 Web 版，内容随前端仓库版本化发布，不建设 CMS；
- 当前处于测试阶段：备份后清空既有账号，所有测试账号均从新流程创建。

# 2. 背景与问题

## 2.1 当前体验问题

当前 Web 版的新角色会进入侠客岛，并由前端提供零散的情境提示。该方式仍存在以下问题：

- 玩家缺少由任务、奖励和解锁组成的明确成长主线；
- 创角 Web 表单只支持账号、密码、名字和性别，无法理解或决定四维；
- 服务端虽保留旧的随机天赋逻辑，但 Gateway 会自动接受，玩家既看不到结果，也无法选择；
- Web 帮助页仅有少量固定主题，采用原始文本展示，缺少搜索、分类、上下文和可执行引导。

## 2.2 可复用的 PKUXKX 资源

PKUXKX 镜像中，实际由登录流程启用的新手体验是：

```text
pkuxkx-utf8/d/newbie_lxsz/
```

而不是较早的 `d/newbie/` 新手学堂。

柳秀山庄新手村共有 62 个文件和 38 个顺序检查点，覆盖状态查看、吃喝、场景探索、NPC 交互、战斗、经济、拜师、学习、武功激发、绝招、阅读和毕业交接。其核心状态为：

```text
newbie_village/quest_index
newbie_village/master
newbie_village/pfm
newbie_village/done
```

PKUXKX 的 `doc/help/` 约有 360 篇帮助资料，覆盖新手、人物、战斗、武功、门派、任务、地图和谜题；其中还含有大量过期、PKU 独有或尚未迁入当前 XKX 的内容，不能直接面向玩家发布。

# 3. 目标、边界与非目标

## 3.1 首发目标

1. 新注册角色可在 Web 中完成柳秀山庄完整新手流程，并毕业进入扬州。
2. 玩家在毕业前理解四维含义，并完成一次合法、不可逆的属性分配。
3. 新手村过程有清晰的当前目标、下一步和完成反馈。
4. Web 帮助中心按玩家阶段展示经审核的资料；首发覆盖新手村、毕业属性、基础系统和扬州入门。
5. 整条流程可通过自动化端到端用例验收。

## 3.2 非目标

- 不整体替换当前 XKX 的核心引擎、战斗、Web、网关或部署体系。
- 不把 PKUXKX 的 `u/`、运行数据、历史备份和玩家数据迁入当前项目。
- 不在首发迁入全部 PKU 地图、门派、任务、比武、国战、地产或高级玩法。
- 不建设帮助内容 CMS、在线编辑后台或用户投稿系统。
- 不收集、验证或保存电子邮箱；密码找回另行设计。
- 不向普通玩家开放文字客户端帮助体验；文本入口仅保留服务端异常恢复用途。

# 5. 柳秀山庄新手村方案

## 5.1 新手村体验结构

柳秀山庄的 38 步流程按体验可分为五段：

| 阶段 | 玩家学习内容 |
|---|---|
| 生存与探索 | 查看自身状态、观察场景、取水、进食、方向移动、攀爬 |
| 叙事与交互 | 打听、给予、跟随、敲门、洗浴、睡觉 |
| 经济与恢复 | 查找地点、取款、购买药物、治疗伤势 |
| 战斗与武功 | 切磋、拜师、学习、装备、激发、准备、练习、绝招 |
| 毕业与交接 | 击杀教学目标、阅读、告别、乘车前往扬州 |

每一步由服务端的任务序号控制，完成时发放教学奖励并给出下一步说明。Web 不应依赖猜测房间文本来推断进度，而应读取服务端发布的当前新手状态。

## 5.2 服务端状态机

在现有 `newbie_village/*` 基础上补充明确的流程状态：

```text
newbie/flow_version = "lxsz-v1"
newbie_village/quest_index
newbie_village/master
newbie_village/pfm
newbie_village/exit_state = learning | attr_pending | graduated
newbie_village/attr_respec_used = 0 | 1
newbie_village/done = 0 | 1
```

状态转换：

```text
创建角色
  → learning：进入未明谷，执行 38 步教学
  → attr_pending：完成离开确认后，等待属性分配
  → graduated：属性确认、清理教学内容、进入扬州
```

设计要求：

- 玩家在 `learning` 或 `attr_pending` 断线后，登录必须回到柳秀山庄；
- `attr_pending` 不得再次结算毕业奖励；
- 仅服务端可从 `attr_pending` 转为 `graduated`；
- 毕业后设置扬州客店为存档出生点，不可再回新手村；
- 不再使用 `xkd/intro_done` 作为新号流程标记。

## 5.3 登录路由

新号的路由优先级调整为：

```text
新注册角色或 lxsz-v1 未毕业角色
→ /d/newbie_lxsz/weiminggu.c

已毕业角色
→ 正常 startroom；首次毕业后为 /d/city/kedian
```

测试阶段清空账号后，所有账号均应从这一流程开始。旧数据仅保留在隔离备份中，不应被测试服务器加载。

## 5.4 教学内容的毕业清理

柳秀山庄本质是教学沙盒。毕业时执行一次原子化结算：

1. 校验毕业状态与属性选择是否合法；
2. 清除柳秀山庄教学武功、教学武器、教学物品和教学货币；
3. 清除教学过程中获得的战斗经验；
4. 写入最终四维；
5. 发放毕业潜能奖励；
6. 写入毕业和存档出生点；
7. 移动至扬州。

若任一步失败，角色必须保持在 `attr_pending`，不得出现“技能已清除但无法毕业”的中间状态。

## 5.5 迁移包与适配项

首批迁入范围：

```text
pkuxkx-utf8/d/newbie_lxsz/         # 62 个场景、NPC、物品和任务逻辑
pkuxkx-utf8/kungfu/skill/taiyi-*.c # 教学用太乙武功
```

已确认需要适配或补充的依赖：

| PKU 依赖 | 当前 XKX 情况 | 处理方式 |
|---|---|---|
| `/clone/misc/guider` | 缺失 | 新建简化路引，或取消毕业发放 |
| `/d/city/obj/yangjing` | 缺失 | 迁入或替换为当前可用药物 |
| `/kungfu/class/shenlong/zhang` | 缺失 | 改为新手村专用 NPC/逻辑，不迁完整神龙体系 |
| 太乙教学武功 | 缺失 | 作为隔离教学包迁入，毕业时统一清除 |
| PKU 原始数值/奖励 | 与当前数值体系可能不同 | 按当前 XKX 规则审查后调整 |

扬州广场、扬州客店、基础货币、睡眠和状态等接口在当前项目中已有对应能力，可优先复用。

# 6. 毕业属性选择方案

## 6.1 规则

教学期间固定四维：

| 膂力 | 悟性 | 根骨 | 身法 |
|---:|---:|---:|---:|
| 20 | 20 | 20 | 20 |

毕业属性分配规则：

```text
str + int + con + dex = 80
10 <= str, int, con, dex <= 30
仅允许确认一次
```

服务端不得信任 Web 传入值；必须重新校验范围、合计、流程状态和一次性标记。

## 6.2 触发时机

玩家完成新手村最后一步并首次确认前往扬州时：

1. 停止最终离开动作；
2. 将状态置为 `attr_pending`；
3. 服务端向 Web 推送属性选择事件；
4. 玩家完成选择并确认；
5. 执行毕业原子化结算；
6. 播放出发叙事并抵达扬州。

## 6.3 Web 界面要求

界面叙事标题：

> 踏入江湖前，定下你的根骨

说明文案：

> 柳秀山庄所学皆为引路之法。离开此地前，你可依心中所向，重定天赋一次。此后不可更改。

界面必须包含：

- 剩余点数和总点数；
- 每项属性的加减控件；
- 初始值与最终值的明确标签；
- 每项属性的可读说明与玩法倾向；
- 预设方案和自定义方案；
- 未满足 80 点时禁用确认；
- 最终不可逆确认弹窗。

属性说明：

| 属性 | 面向玩家的说明 |
|---|---|
| 膂力 | 影响近身攻击的威力与负重能力，适合偏好正面交锋、使用重兵器的侠士。 |
| 悟性 | 影响学习、领悟与研读武学的效率，适合希望更快掌握复杂武功的侠士。 |
| 根骨 | 影响体魄、恢复与内功成长基础，适合重视生存、续航和持久作战的侠士。 |
| 身法 | 影响闪避与行动灵活性，适合轻功、剑法与规避伤害的侠士。 |

建议预设：

| 预设 | 膂力 | 悟性 | 根骨 | 身法 |
|---|---:|---:|---:|---:|
| 均衡 | 20 | 20 | 20 | 20 |
| 剑客 | 18 | 20 | 18 | 24 |
| 苦修 | 18 | 18 | 26 | 18 |
| 求道 | 16 | 28 | 18 | 18 |

界面实现遵循现有 Web UI 规范：

- 使用 `--attr-*`、`--jade`、`--cinnabar` 等既有 token；
- 使用 `--font-display` 作为标题字体、`--font-body` 作为正文；
- 预设使用 `ChoiceRow`，不使用原生下拉框；
- 玩家界面不展示 `str`、`int`、`jifa` 等内部字段或命令名；
- 所有双数值必须带“初始 / 最终”等含义标签。

## 6.4 Web 与服务端协议

建议服务端推送：

```json
{
  "type": "newbie.attribute_select",
  "budget": 80,
  "min": 10,
  "max": 30,
  "initial": { "str": 20, "int": 20, "con": 20, "dex": 20 }
}
```

Web 提交选择后，由服务端专用入口完成确认。为调试和异常恢复保留隐藏文字后备入口，但不在玩家 Web UI 中展示。

# 7. Web 帮助中心重塑方案

## 7.1 内容原则

帮助中心的原则是：

> 只发布当前版本真实可用、对当前阶段玩家有行动价值的内容。

PKUXKX 帮助资料仅作为内部来源。每篇文章需重新校对当前 XKX 的：

- 地图与 NPC 是否存在；
- 门派与任务是否可用；
- 数值、条件与奖励是否一致；
- Web 是否具备对应交互；
- 是否包含已废弃、自动化或不适合首发的内容。

不直接复制大段原文；内部记录原始来源路径，便于追溯事实依据。

## 7.2 首发内容范围

首发只发布以下类别：

| 分类 | 首发主题 |
|---|---|
| 柳秀山庄 | 当前目标、状态、吃喝、探索、NPC 交互、战斗、拜师、毕业 |
| 四维与人物 | 四维属性、气血、精神、内力、精力、潜能、装备 |
| 基础玩法 | 移动、观察、交互、战斗、恢复、学习、武功操作、死亡 |
| 扬州入门 | 扬州生存、基础经济、初期探索、下一步建议 |
| 规则 | 基础行为规则、账号和测试说明 |

以下内容仅整理为内部草稿库，待功能迁入且验证后发布：

- 未迁入门派与对应新手任务；
- 未迁入地图和交通路线；
- 高级任务、比武、国战、地产、婚姻；
- 解谜、宝石、特殊职业、长期养成；
- `oldhelp/`、历史更新、旧客户端和废弃系统。

## 7.3 内容模型

每篇帮助文章使用可版本化的结构化元数据：

```text
id                 稳定主题 ID
title              玩家可见标题
summary            一句话说明
category           分类
stage              新手村 / 初入江湖 / 成长 / 高级
prerequisites      前置状态、等级或位置
body               审核后的正文
related            相关推荐
actions            可执行的 Web 动作
source             PKU 原始来源路径，仅内部可见
status             draft / reviewed / published / deprecated
```

属性选择页、帮助文章、角色面板中的属性说明必须共用同一内容来源，避免规则和文案漂移。

## 7.4 Web 信息架构

### 帮助首页

- 搜索框：支持“怎么拜师”“打不过怎么办”“怎么恢复”等意图和标签；
- 你现在需要知道：按新手村任务阶段推荐；
- 当前任务相关帮助；
- 常用主题；
- 最近查看。

### 文章页

每篇文章按以下顺序组织：

1. 一句话说明；
2. 什么时候需要它；
3. 关键规则；
4. 操作步骤；
5. 风险与常见错误；
6. 可执行动作；
7. 相关阅读。

Web 玩家看到的操作名称应是“查看状态”“向师父请教”“装备兵器”等可读语言；对应内部命令只由按钮处理，不裸露给普通玩家。

### 地图资料

不直接将 PKU ASCII 地图作为主内容。对于当前可玩区域，优先展示可交互区域图、地点卡片和路线；ASCII 图仅可作为补充资料。

## 7.5 发布方式

- 审核后的文章和索引放在前端仓库中，随 Web 构建发布；
- 首发不建设 CMS；
- 每次内容修改经版本控制、测试和审查后发布；
- 未发布内容保持 `draft` 状态，不出现在玩家搜索和推荐结果中；
- 文字客户端的传统帮助不作为产品目标，仅保留服务端兼容能力。

# 9. 验收标准

## 9.1 新手流程

- 新账号注册后直接进入未明谷，不进入侠客岛；
- 可通过 Web 完成柳秀山庄 38 个检查点；

- 断线后可继续当前任务，不丢失进度。

## 9.2 属性流程

- 新号教学期间固定为四维 20；
- 毕业时触发属性选择界面；
- 仅接受总和为 80、单项 10～30 的提交；
- 重复提交、非法提交和非毕业角色提交均被拒绝；
- 确认后不可再通过常规入口修改；
- 断线后待定状态可恢复；
- 毕业后教学技能、教学经验、教学物品和教学货币按规则清除；
- 最终进入扬州，并设置扬州客店为后续出生点。

# 10. 风险、回滚与后续扩展

## 10.1 主要风险

| 风险 | 应对方式 |
|---|---|
| PKU 新手村依赖当前不存在的对象、技能或数值 | 逐文件编译、建立依赖清单、以最小替代实现适配 |
| 教学逻辑与 Web 动作不兼容 | 以 38 步 e2e 覆盖；优先补 Web 操作而非要求玩家输入命令 |
| 属性提交篡改或重复结算 | 所有校验、状态转换和奖励均由服务端执行 |
| 断线导致毕业状态损坏 | 使用 `attr_pending` 持久状态并实现幂等结算 |
| PKU 帮助内容与当前功能不一致 | 采用草稿—审核—发布流程，不直接导入原文 |
| 测试清档影响排查 | 清档前保留隔离备份，并记录恢复步骤 |

## 10.2 回滚策略

- 新手村和属性系统以独立流程版本标记控制；
- 测试期如出现阻断问题，可恢复隔离账号备份并回退代码版本；
- 已进入 `attr_pending` 的账号必须提供服务端修复入口；
- 帮助文章随前端版本发布，可通过回退前端版本或下架文章状态恢复。

## 10.3 后续扩展

当首发流程稳定后，按以下顺序扩展：

1. 当前已可玩区域的扬州帮助与地图；
2. 已迁入门派的门派说明、拜师条件和新手任务；
3. 通用日常任务和成长路线；
4. 新地图、门派、武功与任务迁入时同步发布对应帮助；
